# Kana 이후 N5 단어 vocabulary seed 구현

## 배경

`docs/after_kana.md` 검토 결과, kana 문자 단위 학습 다음 단계는 N5 문법보다 단어 단위 읽기 전환이 우선이라고 판단했다.

현재 코드베이스는 kana 학습도 별도 `words` 도메인 없이 `questions` 테이블에 seed 하는 구조다. 따라서 이번 결정은 기존 `questions`, `session_questions`, SRS, Telegram bot flow 를 재사용하는 작은 단계로 잡았다.

## 변경 사항

- `cmd/ja/vocab_seeder/main.go`
  - N5 단어 100개를 repo-owned seed data 로 추가.
  - 단어당 3개 문항 생성:
    - `kana -> meaning` 객관식
    - `meaning -> kana` fill_blank
    - `meaning -> kana handwriting`
  - 기존 `QuestionRepository.CreateBatch` 로 300개 문항을 batch insert.
- `cmd/ja/vocab_seeder/main_test.go`
  - 문항 생성 metadata, options 중복, explanation 구성, seed data integrity, 300개 생성 수량 검증.
- `internal/external/llm.go`
  - `GradeHandwriting` public signature 유지.
  - 손글씨 prompt 를 단일 kana 전제에서 kana character 또는 short kana word 전제로 일반화.
  - system/user prompt helper 분리.
- `internal/external/llm_test.go`
  - 손글씨 prompt helper 단위 테스트 추가.
- `internal/bot/session_flow.go`
  - fill_blank 안내 문구에서 romaji/영어 전용 표현 제거.
- `README.md`
  - `go run ./cmd/ja/vocab_seeder` seed 실행 절차 문서화.
- `STATUS.md`
  - TODO 링크 추가 및 완료된 TODO 항목 제거, 최근 완료 기록 업데이트.
- `docs/todos/n5_vocab_seed.md`
  - `cmd/ja/vocab_seeder` 신규 구현 plan 작성 및 실행 완료 후 삭제.

## 주요 결정 사항

- **DB Schema**: v0.3 이전까지는 `words`, `kanji`, `word_kanji` 정규화 테이블을 추가하지 않고 기존 `questions` 테이블을 재사용한다.
- **Kanji 처리**: 한자는 정답 채점 대상이 아니라 explanation 보조 노출(`(<b>kanji</b>)` 형태)로만 사용한다.
- **Handwriting**: 기존 `kana_handwriting` type과 Mini App flow를 재사용하되, LLM prompt를 단일 kana에서 짧은 kana 단어까지 허용하도록 일반화했다.
- **Prompt 일반화**: `fill_blank` 안내 문구의 romaji 전용 표현을 vocabulary seed 구현에 맞춰 일반화했다.
- **Weighting**: `session_builder`의 category weighting은 이번 단계에서 변경하지 않았다.

## 미실행 및 수동 작업

- `go run ./cmd/ja/vocab_seeder`는 로컬 DB에 실제 seed row 300개를 insert 하는 명령이라 자동 실행하지 않았다. 재실행 시 중복 insert 되는 기존 seeder 계열 제약이 있으므로, DB 상태를 확인한 뒤 수동 실행이 필요하다.
